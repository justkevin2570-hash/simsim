# CLAUDE.md — 학교 행정업무 도우미 개발 지침

이 문서는 AI 코드 어시스턴트(Claude Code, Cursor 등)가 본 프로젝트에서 일관된 코드를 생성하도록 돕는 지침서입니다. 모든 작업은 이 문서의 규칙을 따라야 합니다.

---

## 1. 프로젝트 개요

- **이름**: 학교 행정업무 도우미 (School Admin Helper)
- **목적**: 초·중·고 교사의 행정업무(공문·계획서·품의서)를 단계별 마법사로 안내하고, 한글(hwp) 파일에 붙여넣을 수 있는 **텍스트 양식**을 자동 생성하는 웹 서비스
- **타겟**: 현직 교사 (특히 신규교사, 부장교사)
- **첫 구현 도메인**: 교육과정 박람회 (운영계획서 / 품의서 / 마무리)

---

## 2. 핵심 원칙

1. **서버리스**: 모든 로직은 클라이언트(브라우저)에서 동작한다. API 라우트, 데이터베이스, 외부 서버 호출을 추가하지 않는다.
2. **개인정보 비저장**: 학생/학부모 정보를 입력받거나 저장하지 않는다. 모든 데이터는 사용자 브라우저의 `localStorage`에만 보관한다.
3. **출력은 텍스트**: 결과물은 항상 사용자가 복사해서 한글에 붙여넣을 수 있는 평문 텍스트다. 파일 다운로드(docx/hwp)는 만들지 않는다.
4. **마법사 패턴**: 모든 업무는 "단계별 입력 → 텍스트 생성 → 복사" 흐름을 따른다.
5. **데이터와 로직 분리**: 선택지·템플릿은 코드와 분리된 데이터 파일(`lib/options.ts`, `lib/templates/`)에 둔다.

---

## 3. 기술 스택 (고정)

| 영역 | 선택 | 비고 |
|------|------|------|
| 프레임워크 | Next.js 16 (App Router) | `src/` 디렉터리 사용하지 않음 |
| 언어 | TypeScript (strict 모드) | `any` 사용 금지 |
| 스타일 | Tailwind CSS | 인라인 CSS 금지 |
| 컴포넌트 | shadcn/ui | 필요할 때만 추가 |
| 상태 | Zustand | 전역 상태 필요 시에만 |
| 폼 | react-hook-form + zod | 검증 필수 |
| 검색 | Fuse.js | 한글 초성은 자체 구현 |
| 저장소 | `localStorage` | 학교명·작성자·이력 |
| 배포 | Vercel | GitHub 연동 |

새 라이브러리 추가는 이 문서에 먼저 반영한 후 설치한다.

## 4. 폴더 구조 규칙

```
app/                # Next.js 페이지·라우팅만
components/         # 재사용 UI 컴포넌트
lib/                # 비즈니스 로직 (UI 의존 없음)
  ├── tasks.ts          # 업무 마스터 데이터
  ├── hangul.ts         # 한글 유틸
  ├── search.ts         # 검색 로직
  ├── store.ts          # Zustand 스토어
  ├── options.ts        # 마법사 선택지
  └── templates/        # 텍스트 양식 생성 함수
```

- `app/` 안에는 UI 코드 외 비즈니스 로직을 두지 않는다.
- `lib/templates/`의 함수는 입력 객체를 받아 **string**을 반환한다. DOM, React, fetch에 의존하지 않는다.

## 5. 코딩 컨벤션

### 5.1 TypeScript

- 모든 함수·컴포넌트에 명시적 타입을 작성한다.
- `type` 위주로 작성하고, 확장이 필요한 경우에만 `interface`를 쓴다.
- `any` 금지. 불가피하면 `unknown` 후 타입 가드 사용.
- 외부에서 쓰는 타입은 같은 파일에서 `export`.

### 5.2 컴포넌트

- 함수형 컴포넌트만 사용한다.
- 클라이언트 전용 컴포넌트는 파일 최상단에 `'use client';`를 명시한다.
- 1 파일 = 1 컴포넌트 원칙. 단, 같은 도메인의 작은 보조 컴포넌트는 같은 파일에 둘 수 있다.
- props는 `type Props = {...}`로 컴포넌트 위에 선언한다.

### 5.3 네이밍

- 컴포넌트: `PascalCase.tsx` (예: `SearchBox.tsx`)
- 유틸/로직: `camelCase.ts` (예: `hangul.ts`)
- 상수: `UPPER_SNAKE_CASE`
- 한국어 식별자 금지. 단, 사용자에게 보이는 문자열은 한국어.

### 5.4 스타일

- Tailwind 유틸리티 클래스 사용. 클래스가 길어지면 의미 단위로 줄바꿈한다.
- 색상은 `gray`, `blue` 계열 위주. 임의의 hex 사용 자제.
- 반응형: `sm:`, `md:`, `lg:` 순서로 작성.

### 5.5 폼 처리

- 입력 검증은 zod 스키마로 정의한다.
- react-hook-form의 `register` / `handleSubmit` / `formState`를 표준으로 쓴다.

## 6. 도메인 모델

### 6.1 업무(Task)

```ts
type Task = {
  id: string;
  title: string;
  category: string;
  tags: string[];        // 검색용 별칭
  description: string;
  icon: string;          // 이모지 1자
  path: string;          // Next.js 라우팅 경로
};
```

업무를 추가할 때는 `lib/tasks.ts`에만 등록한다. 라우팅·검색·메뉴가 자동으로 따라간다.

### 6.2 마법사(Wizard)

- 각 마법사는 `app/<domain>/<task>/page.tsx`에 위치한다.
- 페이지는 (a) 단계 정의, (b) Zustand 상태, (c) 진행 UI, (d) 결과 화면으로 구성한다.
- 단계 컴포넌트는 `components/wizards/<Domain><Task>/Step<N>.tsx`로 분리한다.

### 6.3 템플릿 함수

```ts
// lib/templates/fairPlan.ts
export type FairPlanInput = { /* 필드 */ };
export function generateFairPlanText(input: FairPlanInput): string { /* ... */ }
```

- 입력 타입과 생성 함수를 한 파일에서 함께 export 한다.
- 한글 문서 양식은 **공공기관 표준 양식(개조식: 가/나/다)** 을 따른다.
- 함수 내부에서 외부 I/O, 랜덤, 현재 시각 의존을 피한다(테스트 가능성).

## 7. 검색 기능 규칙

- 메인 페이지(`app/page.tsx`)는 검색 중심 UI다.
- `Fuse.js` 옵션의 `threshold`는 `0.4`를 기준값으로 시작한다.
- 한글 초성 검색은 `lib/hangul.ts`의 `getChoseong`을 사용한다.
- 입력은 100ms 디바운싱한다.
- 키보드 조작(↑/↓/Enter/Esc)을 반드시 지원한다.
- 결과는 최대 8개로 제한한다.

---

## 8. UI/UX 원칙

- **첫 화면은 검색창 하나**. 시각적 노이즈를 최소화한다.
- 마법사는 **한 화면에 한 가지 질문**을 원칙으로 한다.
- 진행률 표시(예: `2 / 6 단계`)를 항상 보여준다.
- 결과 화면에는 항상 **복사 버튼**과 **수정하기 버튼**을 둔다.
- 에러 메시지는 한국어로, 정중한 어조로 작성한다.
- 모바일 대응은 1차 출시 후 진행한다(현재는 데스크탑 우선).

## 9. 작업 시 따라야 할 절차

AI 어시스턴트가 코드를 작성·수정할 때:

1. **변경 전 계획 제시**: "이 작업을 위해 다음 파일을 수정/생성합니다" 목록을 먼저 보여준다.
2. **타입 우선**: 새 기능은 타입 정의 → 함수 → UI 순서로 작성한다.
3. **데이터 분리 유지**: 화면 코드에 선택지·문구를 하드코딩하지 않는다.
4. **기존 패턴 따르기**: 이미 있는 마법사·템플릿 구조를 그대로 모방한다.
5. **불필요한 의존성 추가 금지**: 라이브러리 추가가 필요하면 사용자에게 먼저 확인한다.
6. **주석은 한국어**: 비즈니스 로직 주석은 한국어로, 기술적 TODO만 영어 허용.

## 11. 향후 확장 가이드

새 업무 추가 시:

1. `lib/tasks.ts`에 Task 등록
2. `lib/options.ts`에 마법사 선택지 추가
3. `lib/templates/<도메인>.ts`에 텍스트 생성 함수 추가
4. `app/<도메인>/<업무>/page.tsx`에 마법사 페이지 작성
5. 필요 시 `components/wizards/<도메인>/`에 단계 컴포넌트 분리

새 도메인(예: 현장체험학습) 추가 시 위 1~5를 반복한다.
